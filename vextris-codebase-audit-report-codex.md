## 1. System Architecture Overview

Vextris is a small Vite and TypeScript browser game with a canvas renderer and a deterministic falling-block engine. Its architecture is straightforward:

- Browser entry and real-time loop live in `public/main.ts`.
- Core mutable game state and gameplay orchestration live in `src/engine/gameLoop.ts`.
- Shared state types live in `src/engine/types.ts`.
- Board representation and row operations live in `src/engine/board.ts`.
- Tetromino definitions, collision, wall kicks, and ghost-piece projection live in `src/engine/pieces.ts`.
- Rotation-specific active-piece mutation lives in `src/engine/rotation.ts`.
- Deterministic RNG and 7-bag generation live in `src/engine/random.ts`.
- Vex mechanics are split across focused modules and re-exported from `src/engine/vex.ts`.
- Canvas drawing and DOM stat rendering live in `src/render/canvasRenderer.ts`.
- Sound hooks live in `src/audio/audioManager.ts`.
- Tunable rules and constants live in `src/config/gameConfig.ts`.

The project has a strong engine/test center of gravity. Gameplay logic is mostly independent from browser APIs, which is why the test suite can exercise board state, piece movement, line clears, RNG reproducibility, rotation, and Vex behavior without launching the browser.

The runtime starts in `public/main.ts`. It looks up DOM elements, creates a `GameState`, starts the game, handles keyboard events, runs `requestAnimationFrame`, calls `tick`, handles held-key movement, and renders every frame. The entry file imports engine commands from `src/engine/gameLoop.ts`, rendering from `src/render/canvasRenderer.ts`, and audio hooks from `src/audio/audioManager.ts`.

The game uses one mutable `GameState` object. Engine functions mutate that object directly. This keeps the code compact and easy to test, but it also means timing, RNG, board mutation, score changes, active piece movement, spell bank changes, and status transitions are all coupled through a shared object.

## 3. Major Gameplay Execution Paths

### Startup and First Spawn

1. `public/main.ts` creates a random seed.
2. `createGameState(seed)` creates a READY `GameState` with empty board, empty queues, initial score/level/timers, RNG state, and spell IDs.
3. `startGame(state)` transitions READY to PLAYING.
4. `startGame` reconstructs RNG from state and calls `spawnPiece`.
5. `spawnPiece` fills the next queue from the 7-bag, shifts one shape out, creates an active piece at its spawn origin, maybe attaches a Vex mark, checks spawn collision, sets ghost piece, resets lock timers, and saves RNG state.
6. The browser loop begins calling `gameLoop`.

### Frame Loop

1. `gameLoop(now)` computes `deltaMs`.
2. If status is PLAYING, it calls `tick(state, deltaMs)`.
3. It handles held-key movement and soft drop.
4. It calls `render(...)` every frame regardless of status.
5. It schedules the next animation frame.

### Gravity and Lock Delay

1. `tick` increments `gameTick`.
2. It gets the gravity interval from current level.
3. It accumulates `gravityTimerMs`.
4. While enough time has accumulated, it attempts to move the active piece down.
5. If the piece cannot move down, gravity movement stops.
6. It checks whether the piece is grounded.
7. If grounded, it accumulates `lockDelayTimerMs`.
8. Once lock delay reaches `LOCK_DELAY_MS`, it calls `lockAndResolve`.
9. If not grounded, lock delay is reset.

### Movement

- `moveLeft` and `moveRight` check PLAYING status, calculate a new origin, run `collides`, mutate origin on success, rebuild ghost, and reset lock if grounded.
- `softDrop` checks PLAYING status, attempts one-row downward movement, rebuilds ghost, and awards soft-drop score on success.
- `hardDrop` computes ghost position, moves active piece directly there, awards hard-drop score, and calls `lockAndResolve` immediately.

### Rotation

1. `rotateCW` or `rotateCCW` validates active piece and PLAYING status.
2. `tryRotate` determines target rotation state and kick list.
3. It records the old Vex mark world position if present.
4. It tries kick origins until a non-colliding placement is found.
5. On success, it mutates active piece origin, rotation state, and blocks.
6. It remaps Vex mark block index.
7. It rebuilds ghost piece.
8. It resets lock delay if grounded and under reset limit.

### Lock and Resolve

1. `lockAndResolve` writes active piece blocks into the board.
2. If the active piece had a Vex mark, the mark is transferred to the locked cell.
3. Completed visible rows are found.
4. If rows exist, it plays line-clear sound, clears rows, collapses columns, scores, and increments combo.
5. If no rows clear, combo resets.
6. Vex alignment detection runs after line clear/collapse.
7. Alignment grants add score and audio feedback.
8. Spell selection is initialized if the bank was previously unselected.
9. Total lines and level update.
10. Active and ghost pieces are cleared, timers reset.
11. A new piece is spawned.
12. If spawn fails, status becomes GAME_OVER.

### Vex Alignment

1. `findVexAlignments` exits early if spell bank is full.
2. It scans visible rows only.
3. For each occupied marked cell, it checks 8 adjacent directions.
4. If another occupied marked visible cell is found, it optionally consumes both marks.
5. It grants a random Vex spell.
6. It stops after max grants per lock or full bank.

### Vex Casting

1. `castSelectedSpell` rejects non-playing states, active casting, empty bank, or ineligible spell.
2. It sets status to CASTING.
3. It reconstructs RNG.
4. For COLOR:
   - selects a present color weighted by occupied cell count.
   - destroys all matching locked cells.
   - collapses columns.
   - resolves post-cast line clears.
5. For SHAPE:
   - selects a present shape uniformly.
   - destroys all matching locked cells.
   - collapses columns.
   - resolves post-cast line clears.
6. For SHADOW:
   - inverts visible empty/filled space into shadow blocks.
   - awards Shadow score.
7. It awards destroyed-cell score for Color/Shape.
8. It removes the spell and updates selected index.
9. It attempts to shift the active piece upward if the cast caused collision.
10. It rebuilds ghost if needed, resets lock delay if grounded, saves RNG, and restores previous status.

## 4. State Management Analysis

### State Shape

`GameState` includes:

- Board state.
- Active and ghost piece.
- Next queue.
- Spell bank and selected spell index.
- Score, level, lines, combo.
- Status.
- Gravity and lock timers.
- Game tick.
- RNG seed/state.
- Current 7-bag and bag index.
- Next spell ID.

This state model is compact and complete enough to support deterministic tests. The `rngState`, `pieceBag`, `pieceBagIndex`, and `nextSpellId` fields are particularly important for reproducibility.

### Mutation Model

All gameplay commands mutate `GameState` in place. This is simple and fast for a small browser game. It also means call ordering is crucial. For example, line clears must happen before Vex alignment detection, and RNG must be saved after every operation that draws random values.

The board itself is a mutable `Cell[][]`. Board helper functions mutate directly. Tests are therefore essential for maintaining invariants.

### Strengths

- Deterministic seeded state enables strong reproducibility tests.
- Board coordinate conventions are centralized and well tested.
- Game state contains enough runtime fields to avoid hidden global gameplay state, except `pieceIdCounter`.
- Most engine behavior is browser-independent.

### Weaknesses

- `pieceIdCounter` is module-global and not part of `GameState`; this could affect exact replay/snapshot expectations if piece IDs ever become semantically important.
- Audio side effects are called directly from engine functions, reducing engine purity.
- Input repeat timers live outside `GameState` in `public/main.ts`.
- CASTING is a synchronous status flip rather than a real cast-state timeline.
- Some config values describe state behavior that is not actually represented in `GameState`, such as input buffer and cast buffer state.

## 5. Technical Debt Assessment

### High-Impact Debt

`public/main.ts` is not included in the current TypeScript project. `tsconfig.json` has `rootDir: "src"` and `include: ["src/**/*.ts"]`, so browser entry code can drift without `tsc --noEmit` catching it.

`src/engine/input.ts` is a scaffold only, but input is central to gameplay and currently lives in `public/main.ts`. This creates a mismatch between the intended architecture and the implementation.

Several timing constants are duplicated or unused. `public/main.ts` defines `DAS_DELAY = 167` and `ARR_INTERVAL = 33`, while `gameConfig.ts` also exports `DAS_MS` and `ARR_MS`. `SOFT_DROP_INTERVAL_MS`, `INPUT_BUFFER_WINDOW_MS`, cast buffer constants, and Shadow darken timing are not fully implemented.

`src/engine/vex.ts` is a barrel but still imports many implementation symbols it does not use. This likely came from refactoring a monolithic file into focused modules.

`src/audio/audioManager.ts` has mute state, but `playSound` always logs and does not check `isMuted`.

### Medium-Impact Debt

`rotation.ts` imports types from `gameLoop.ts`, while `gameLoop.ts` re-exports rotation functions. The comment says `types.ts` was extracted to break circular dependencies, but `rotation.ts` still couples to the gameplay barrel instead of importing types directly from `types.ts`.

`gameLoop.ts` remains the largest source module at roughly 590 lines. It is manageable but still owns many responsibilities: state factory, queue, spawning, movement, gravity, lock, scoring, leveling, Vex orchestration, and casting.

The renderer mixes canvas drawing and DOM updates. `renderStats` queries DOM every frame, and `renderSpellBank` rebuilds `innerHTML` every frame.

The documentation has stale references. `UPGRADE_ROADMAP.md` says older line counts and file roles in some places, while the current code has already been split.

### Low-Impact Debt

There are many tests with unused local variables and imports under stricter compiler settings.

Some exported renderer helper functions may exist mainly for future tests or manual reuse. That is acceptable, but if not intentional they increase API surface.

## 6. Bug Risk Assessment

### Input Repeat Risks

`Space` triggers `hardDrop` on every keydown event. Browsers fire repeated keydown events while a key is held. This can lock multiple pieces if the player holds Space.

Rotation comments say rotation is one-shot, but keydown repeat can also repeat ArrowUp or KeyZ unless `e.repeat` is checked.

Soft drop is performed once per animation frame while ArrowDown is held. This makes soft-drop speed frame-rate dependent and ignores `SOFT_DROP_INTERVAL_MS`.

The code tracks `upHeld` and `zHeld` in `handleHeldKeys`, but does not use them.

### Sound Feedback Risks

`public/main.ts` plays movement, rotation, soft-drop, and hard-drop sounds without checking whether the underlying action succeeded. This can produce misleading audio when moves are blocked.

Engine functions also call `playSound` for gameplay events. Audio is therefore split between input layer and engine layer.

### Cast and Collision Risks

`castSelectedSpell` sets status to CASTING but resolves the cast synchronously and immediately restores previous status. If the design expects animation timing or frozen input timers, that behavior is not truly implemented.

The post-cast upward shift loop uses `for (let shift = 0; shift < 2; shift++)`, which tries only shifts 0 and 1. `CAST_UPWARD_SHIFT_MAX` is 2, and a reader would expect shifts up to 2. The constant is imported but unused.

If a post-cast collision remains after attempted shifts, the code does not explicitly handle it as game over, lock, or denied cast. It just proceeds.

### Gameplay Rule Risks

`collapseColumns` is column-gravity, not standard Tetris row gravity. This is probably intended by Vextris rules, but it is a high-risk area because many people will assume row collapse.

Shadow Vex creates a very large number of shadow blocks after inversion. This is tested, but future rule changes could have large balance effects.

Line clears after Vex casts are resolved inside `vexCasting.ts`, but scoring for those post-cast line clears is not visibly integrated with normal line-clear score in `gameLoop.ts`. That may be intentional, but it is a gameplay-design risk.

### Type/Build Risks

The browser entry point is not typechecked by the default TypeScript pass.

Using `innerHTML` for spell bank is currently low risk because spell types are internal enums, but it is still less robust than DOM node creation.

## 8. Refactor Regression Risks

### Coordinate System

The board uses array rows `0..21`, with visible logical rows `0..19` mapped to array rows `2..21`. Any refactor that touches board coordinates, spawn positions, collision, rendering, or Vex alignment can easily introduce off-by-one bugs. Existing board and piece tests are strong protection.

### RNG Determinism

The game reconstructs RNG from `rngSeed` and `rngState`. Every random draw must save state afterward. Refactoring spawn, Vex mark creation, spell grants, or cast targeting can change deterministic sequences and break reproducibility.

### Vex Mark Rotation

Vex mark remapping has custom behavior for O-piece and nearest-world-cell behavior for other pieces. This area already had historical bugs per comments/tests. Treat it as fragile and keep rotation tests close.

### Lock Delay and Reset Limit

Movement and rotation reset lock delay only when grounded and under `LOCK_RESET_LIMIT`. This behavior is duplicated in `gameLoop.ts` and `rotation.ts`. Refactoring could create inconsistencies.

### Column Collapse

`collapseColumns` preserves top-to-bottom order within each column and drops cells independently. This is central to Vextris behavior and affects line clears, Vex casts, Shadow Vex, and tests.

### Spell Bank Index

Post-cast index behavior is carefully specified: remove selected spell, preserve same logical position when possible, clamp to last item, or set `-1` when empty. Any UI or spell-bank refactor must preserve this.

### Renderer Coupling

Renderer imports types via `gameLoop.ts`, not directly from `types.ts`. Refactoring barrels or type exports can break renderer even if engine behavior is unchanged.

## 9. Gameplay Mechanics Documentation

### Board

- Board is 10 columns by 22 total rows.
- Top 2 rows are hidden spawn rows.
- Visible board is 10 by 20.
- Cells store occupancy plus optional color, shape, and Vex mark.
- `logicalToArrayRow(y)` maps logical y to array row by adding hidden rows.

### Pieces

- Playable pieces are I, O, T, S, Z, J, L.
- `SHADOW` is a pseudo-shape used by Shadow Vex blocks.
- Shapes have four rotation states.
- Spawn origins are defined per shape.
- Wall kicks are defined for J/L/S/Z/T and I; O uses identity kick.
- Collision checks total board bounds and locked-cell overlap.

### Queue and RNG

- RNG is seeded with a string using FNV-1a and Mulberry32.
- The 7-bag contains one of each playable piece and is shuffled by seeded RNG.
- `nextQueue` is maintained at `NEXT_QUEUE_SIZE`.

### Gravity and Lock

- Gravity interval depends on level.
- Level increases every 10 lines.
- Lock delay is 500ms.
- Grounded movement/rotation can reset lock delay up to a configured limit.
- Hard drop bypasses lock delay.

### Scoring

- Line clears use classic single/double/triple/quad base scores multiplied by level.
- Combo bonus is added before multiplying by level.
- Soft drop awards 1 point per cell.
- Hard drop awards 2 points per cell.
- Vex alignment grants score.
- Vex cell destruction grants score.
- Shadow Vex grants score.

### Vex Marks

- Spawned pieces may receive a single Vex mark.
- When the piece locks, the mark transfers to the corresponding locked cell.
- Completed row clears remove any marks in cleared cells.
- Adjacent visible locked marked cells create alignments.
- Alignments grant spells and consume marks when configured.

### Spell Bank

- Spell types are COLOR, SHAPE, and SHADOW.
- Spell bank has a max size.
- `C` cycles selected spell.
- `V` casts selected spell.
- Removing a spell adjusts selected index according to post-cast rules.

### Color Vex

- Eligible when the selected spell is COLOR and the board has occupied cells.
- Chooses a target color weighted by count of occupied cells of each color.
- Destroys all locked cells of that color across the total board.
- Collapses columns and resolves post-Vex line clears.

### Shape Vex

- Eligible when selected spell is SHAPE and the board has occupied cells.
- Chooses a target shape uniformly from shapes present.
- Destroys all locked cells of that shape across the total board.
- Collapses columns and resolves post-Vex line clears.

### Shadow Vex

- Eligible only when visible occupied count is at least 80 cells, or 40% of visible board.
- Clears visible locked cells.
- Creates shadow blocks in spaces that were previously empty visible cells.
- Hidden rows are preserved.
- Shadow blocks become normal locked cells with `colorId = "shadow"` and `shapeId = "SHADOW"`.

### Rendering

- Main board uses 32px cells.
- Renderer draws grid, locked cells, Vex marks, ghost piece, active piece, next preview, stats, and spell bank.
- Spell bank uses inline HTML strings.
- Stats are written to DOM every frame.

## Prioritized Top 10 Highest-Value Improvements

1. Include `public/main.ts` in typechecking.

   The browser entry point owns important gameplay behavior but is excluded by current `tsconfig.json`. Add a separate app tsconfig or adjust project layout so entry code is typechecked.

2. Move input handling into `src/engine/input.ts`.

   The file exists as a scaffold, while real input logic lives in `public/main.ts`. Centralizing input would make DAS/ARR, buffering, repeat guards, and tests much easier.

3. Fix key repeat hazards for hard drop and rotation.

   Guard `Space`, `ArrowUp`, and `KeyZ` using `e.repeat` or explicit release tracking so held keys do not repeatedly hard-drop or rotate.

4. Make soft drop timer-based.

   Use `SOFT_DROP_INTERVAL_MS` instead of dropping once per animation frame. This will make soft-drop speed consistent across machines.

5. Replace duplicated input timing literals with config constants.

   `public/main.ts` duplicates DAS and ARR values already exported by `gameConfig.ts`. One source of truth will reduce tuning errors.

6. Clean unused imports and enable stricter unused checks.

   A strict compiler pass found many stale imports after refactors. Cleaning them up will make future refactors safer.

7. Implement real cast timing or remove pretend async cast state.

   CASTING currently resolves synchronously. Either implement cast animation/timer state properly or simplify the status/config model to match actual behavior.

8. Honor `CAST_UPWARD_SHIFT_MAX` and handle unresolved collision after Vex casts.

   The current loop uses a literal and likely does not test the full intended range. It should define what happens if a cast leaves the active piece colliding.

9. Make audio mute-aware and move success/failure sounds behind action results.

   `playSound` ignores mute, and input code plays sounds even when movement fails. This is a small change with immediate polish impact.

10. Update docs and roadmap to match current code.

   `UPGRADE_ROADMAP.md` and spec-adjacent docs contain stale module descriptions and counts. Refreshing them would reduce onboarding friction.

## Closing Assessment

Vextris is in good shape for a compact game project. The engine is deterministic, well tested, and mostly separated from browser rendering. The main architectural gap is that runtime input and browser entry behavior have not caught up with the cleaner engine/module split. The main bug-risk cluster is input repeat/timing and the partially implemented cast timing model. The highest-value work is not a rewrite; it is tightening the boundaries that already exist.
